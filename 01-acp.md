# ACP (Agent Commerce Protocol) 深度解析

## 一句话定义

ACP 是 OpenAI 与 Stripe 于 2025 年初联合推出的开放标准协议，专注于为 AI 智能体提供标准化的**支付结账能力**。它让 AI 能够安全地代表用户完成商品和服务的购买，实现"对话即交易"的体验。

---

## 为什么需要 ACP？

### AI 支付的三大难题

当 AI 想要帮用户买东西时，面临根本性挑战：

1. **信任问题**：用户如何让 AI 安全地"花钱"而不被滥用？
2. **授权问题**：如何在不暴露敏感支付信息的情况下完成交易？
3. **标准问题**：每个商家支付接口不同，AI 无法统一对接

### ACP 的解决思路

**"安全令牌 + 范围授权"** —— 用户一次性授权 AI 在特定范围内支付，AI 使用临时令牌完成交易，无需接触真实支付信息。

```
传统方式：                      ACP 方式：
┌─────────┐                    ┌─────────┐
│   AI    │                    │   AI    │
└────┬────┘                    └────┬────┘
     │                             │
     │ 需要保存用户信用卡          │ 使用临时令牌
     │ （安全风险）                │ （安全隔离）
     ▼                             ▼
┌─────────┐                    ┌─────────┐
│ 商家API │                    │   ACP   │
└─────────┘                    └────┬────┘
                                    │
                              ┌─────┴─────┐
                              │  Stripe   │
                              │  处理支付  │
                              └───────────┘
```

## 核心架构：四方信任模型

ACP 明确定义了四个角色，构建了完整的信任链条：

```
┌─────────────────────────────────────────────────────────────┐
│                 ACP 四方信任模型                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   👤 买家 (Buyer)                                          │
│   ├─ 人类用户，拥有支付能力                                 │
│   ├─ 授权 AI 代理购买（设定范围和限额）                     │
│   └─ 确认交易并接收商品                                     │
│            │                                                │
│            │ 1. 委托购买 + 授权支付范围                      │
│            ▼                                                │
│   🤖 AI 智能体 (Agent)                                      │
│   ├─ 理解用户购买需求                                       │
│   ├─ 发现商品（通过商家 API 或搜索）                        │
│   ├─ 发起结账（携带用户授权令牌）                           │
│   └─ 向用户汇报交易结果                                     │
│            │                                                │
│            │ 2. 发起结账请求                                 │
│            ▼                                                │
│   🏪 商家 (Seller/Merchant)                                 │
│   ├─ 提供商品/服务信息                                      │
│   ├─ 处理 ACP 结账请求                                      │
│   ├─ 调用 Stripe 完成支付                                   │
│   └─ 履行订单（发货/提供服务）                              │
│            │                                                │
│            │ 3. 处理支付                                     │
│            ▼                                                │
│   💳 支付服务商 (PSP - Payment Service Provider)            │
│   ├─ Stripe（ACP 主要合作伙伴）                             │
│   ├─ 验证支付令牌                                           │
│   ├─ 处理资金结算                                           │
│   └─ 提供风控和合规保障                                     │
│                                                             │
│   信任链条：Buyer 信任 Agent → Agent 通过 ACP 与 Seller 交易 │
│            → Seller 通过 Stripe 获得资金保障               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 各角色深度解析

| 角色 | 核心职责 | 技术实现 | 关键约束 |
|------|----------|----------|----------|
| **Buyer** | 授权 AI、确认交易、接收商品 | 数字钱包、身份验证 | 可随时撤销授权 |
| **Agent** | 理解需求、发现商品、执行交易 | LLM + ACP Client SDK | 只能在授权范围内操作 |
| **Seller** | 商品管理、订单处理、履约交付 | ACP Server 实现 | 需集成 Stripe |
| **PSP** | 支付处理、资金结算、风控 | Stripe API | 负责资金安全和合规 |

## 核心技术：授权令牌机制

ACP 的核心创新是**范围授权（Scoped Authorization）**，它解决了"如何让 AI 安全地花钱"这一根本问题。

### 传统授权 vs ACP 授权

```
传统方式（危险）：
用户 ──信用卡信息──► AI ──信用卡信息──► 商家
         ↑                              ↑
    AI 保存敏感信息              商家接触敏感信息
    （泄露风险）                 （泄露风险）

ACP 方式（安全）：
用户 ──授权范围──► AI ──临时令牌──► 商家 ──令牌──► Stripe
       （"最多花$100"）              （一次性）      （验证并扣款）
         ↑                              ↑
    不暴露真实支付信息           不接触真实支付信息
```

### 授权令牌的工作流程

```
┌─────────────────────────────────────────────────────────────┐
│              ACP 授权令牌生命周期                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  阶段 1：授权创建（Authorization Creation）                  │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  用户通过 OpenAI/Stripe 设置授权                     │    │
│  │  • 授权范围：商品类别、商家、金额上限                │    │
│  │  • 有效期：单次 / 时间段 / 持续                      │    │
│  │  • 示例："允许 ChatGPT 帮我买书籍，最多 $50/月"      │    │
│  └──────────────────┬──────────────────────────────────┘    │
│                     │                                        │
│                     ▼                                        │
│  阶段 2：令牌生成（Token Generation）                        │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Stripe 生成授权令牌（Authorization Token）          │    │
│  │  • 包含授权范围和限制                                │    │
│  │  • 绑定到用户支付方式                                │    │
│  │  • 可撤销、可审计                                    │    │
│  └──────────────────┬──────────────────────────────────┘    │
│                     │                                        │
│                     ▼                                        │
│  阶段 3：令牌使用（Token Usage）                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  AI 在结账时提交令牌                                 │    │
│  │  • 商家验证令牌有效性                                │    │
│  │  • Stripe 验证授权范围                               │    │
│  │  • 在范围内完成扣款                                  │    │
│  └──────────────────┬──────────────────────────────────┘    │
│                     │                                        │
│                     ▼                                        │
│  阶段 4：令牌销毁（Token Destruction）                       │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  交易完成后                                          │    │
│  │  • 单次使用令牌：立即失效                            │    │
│  │  • 多次使用令牌：扣除已用额度                        │    │
│  │  • 用户可随时撤销未使用的授权                        │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 技术规范

### API 设计

ACP 采用 RESTful API 设计，简洁直观：

```yaml
# ACP 核心端点

# 1. 创建结账会话
POST /v1/checkout/sessions
Request:
  {
    "line_items": [{
      "product_id": "book-123",
      "quantity": 1
    }],
    "authorization_token": "acp_token_xxx",
    "customer_id": "user_456"
  }

Response:
  {
    "session_id": "sess_789",
    "status": "requires_confirmation",
    "amount_total": 1599,  # 单位：分
    "currency": "usd"
  }

# 2. 确认并支付
POST /v1/checkout/sessions/{id}/confirm
Response:
  {
    "status": "succeeded",
    "payment_intent_id": "pi_abc",
    "order_id": "order_xyz"
  }

# 3. 查询订单状态
GET /v1/orders/{order_id}
Response:
  {
    "id": "order_xyz",
    "status": "shipped",
    "tracking_number": "1Z999AA..."
  }
```

### 安全机制

| 机制 | 说明 |
|------|------|
| **令牌加密** | 授权令牌使用 JWT 格式，签名防篡改 |
| **范围限制** | 令牌包含明确的金额上限、商品类别限制 |
| **时效控制** | 支持单次使用、时间窗口、持续授权三种模式 |
| **撤销机制** | 用户可随时撤销未使用的授权 |
| **审计日志** | 所有交易可追踪、可审计 |

## ACP vs UCP：支付与商务的分工

| 维度 | ACP | UCP |
|------|-----|-----|
| **核心定位** | 支付结账协议 | 完整商务协议 |
| **覆盖范围** | 购物车 → 支付 → 订单确认 | 发现 → 选择 → 购买 → 履约 → 售后 |
| **推动方** | OpenAI + Stripe | Google |
| **技术风格** | REST API，简洁专注 | 三层架构，功能全面 |
| **搜索能力** | 依赖外部搜索 | 内置语义搜索 |
| **商家接入成本** | 低（只需支付接口） | 高（完整商务能力） |
| **最佳场景** | 已知商品快速结账 | 复杂购物决策 |

### 如何选择？

```
用户场景："帮我买这本书"

场景 A：用户明确知道要买什么
         │
         ▼
    "买《三体》paperback 版"
         │
    ┌────┴────┐
    │         │
    ▼         ▼
   ACP       UCP
（更合适） （也可以）
理由：简单支付，ACP 更轻量

场景 B：用户需要 AI 帮助决策
         │
         ▼
    "推荐一本好的科幻小说"
         │
    ┌────┴────┐
    │         │
    ▼         ▼
   ACP       UCP
（不够用） （更合适）
理由：需要搜索+对比，UCP 更全面
```

## 深入洞察

### ACP 的战略价值

1. **Stripe 的 AI 支付入口**：通过 ACP，Stripe 成为 AI 商务的"默认支付基础设施"
2. **OpenAI 的商务闭环**：ChatGPT 从"对话"延伸到"交易"，形成完整用户体验
3. **商家低门槛接入**：相比 UCP，ACP 接入成本更低，适合快速启动

### 与 Stripe 生态的深度整合

ACP 不是独立协议，而是深度绑定 Stripe：

```
ACP 交易流程中的 Stripe：

1. 授权令牌生成 ──► Stripe Identity（身份验证）
2. 支付处理 ───────► Stripe Payments（支付核心）
3. 风控检查 ───────► Stripe Radar（欺诈检测）
4. 资金结算 ───────► Stripe Treasury（资金管理）
5. 税务合规 ───────► Stripe Tax（自动计税）

商家收益：一键获得 Stripe 全套能力
潜在风险：过度依赖单一支付服务商
```

### 局限性与挑战

1. **生态锁定**：深度依赖 Stripe，对其他 PSP 支持有限
2. **搜索能力弱**：ACP 本身不提供商品搜索，需依赖外部系统
3. **覆盖范围窄**：只解决支付，不解决发现、履约等环节
4. **竞争压力**：面临 UCP、AP2 等协议的竞争

## 实际应用场景

### 场景 1：ChatGPT 购物

```
用户："帮我订一张明天去纽约的机票"

ChatGPT：
1. 调用航班搜索 API（非 ACP）
2. 找到合适航班：$299
3. 检查用户 ACP 授权：额度充足
4. 发起 ACP 结账
5. 用户确认
6. 完成支付
7. 返回电子机票

ACP 作用：安全完成支付环节
```

### 场景 2：AI 客服自动退款

```
用户："这本书质量有问题，我要退货"

AI 客服：
1. 验证订单（通过 ACP 查询）
2. 确认符合退款政策
3. 发起 ACP 退款请求
4. Stripe 处理退款
5. 通知用户退款已到账

ACP 作用：支持逆向交易（退款）
```

## 总结

ACP 代表了"AI 支付"的标准化方向：

- **安全**：通过范围授权和临时令牌，实现"可用但不可见"的支付体验
- **简洁**：专注支付这一核心环节，不试图解决所有问题
- **开放**：虽然是 OpenAI+Stripe 主导，但协议本身开放

它是 AI 商务的"支付基础设施"，与 UCP（商务基础设施）、MCP（工具基础设施）形成互补，共同构建 AI 时代的商业生态。
