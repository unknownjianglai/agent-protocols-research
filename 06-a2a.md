# A2A (Agent2Agent Protocol) 深度调研报告

## 概述

**Agent2Agent Protocol (A2A)** 是由 **Google** 推出的开放协议，旨在实现AI智能体之间的标准化通信和协作。A2A解决了不同厂商、不同平台开发的AI智能体如何互相发现、通信和协作的关键问题。

> **发布时间**: 2025年  
> **主要推动方**: Google  
> **协议类型**: 智能体间通信协议  
> **开源状态**: 完全开源 ([GitHub](https://github.com/a2aproject/A2A))

---

## 核心愿景

A2A的愿景是创建一个**多智能体互联的世界**，让来自不同厂商、不同平台的AI智能体能够：

1. **互相发现** - 找到能够完成特定任务的智能体
2. **安全通信** - 建立可信的通信通道
3. **协作完成任务** - 分工合作完成复杂任务
4. **保持用户控制** - 用户始终掌握最终决策权

## 核心概念

### 1. Agent Card (智能体卡片)

Agent Card是智能体的"名片"，描述了一个智能体的能力和如何连接它：

```json
{
  "name": "TravelBookingAgent",
  "description": "帮助用户预订机票和酒店的智能体",
  "url": "https://travel-agent.example.com/a2a",
  "capabilities": {
    "streaming": true,
    "pushNotifications": false
  },
  "skills": [
    {
      "id": "flight_booking",
      "name": "机票预订",
      "description": "搜索和预订航班",
      "parameters": {
        "origin": "string",
        "destination": "string",
        "date": "date"
      }
    },
    {
      "id": "hotel_booking",
      "name": "酒店预订",
      "description": "搜索和预订酒店",
      "parameters": {
        "location": "string",
        "check_in": "date",
        "check_out": "date"
      }
    }
  ],
  "authentication": {
    "type": "oauth2",
    "authorizationUrl": "https://auth.example.com/oauth/authorize"
  }
}
```

### 2. Task (任务)

Task是A2A中的工作单元，代表一个需要完成的请求：

```json
{
  "id": "task-123",
  "status": "in_progress",
  "created_at": "2025-03-18T10:00:00Z",
  "updated_at": "2025-03-18T10:05:00Z",
  "sessionId": "session-456",
  "messages": [
    {
      "role": "user",
      "content": "帮我预订明天从北京到上海的航班"
    }
  ],
  "artifacts": [],
  "history": []
}
```

### 3. Artifact (产出物)

Artifact是任务执行的结果，可以是文本、文件或结构化数据：

```json
{
  "type": "file",
  "name": "flight_options.pdf",
  "mimeType": "application/pdf",
  "content": "base64encoded...",
  "metadata": {
    "flights_count": 5,
    "price_range": "800-1500"
  }
}
```


### 核心API

```http
# 1. 获取Agent Card
GET /.well-known/agent.json

# 2. 发送任务
POST /tasks/send
{
  "id": "task-123",
  "sessionId": "session-456",
  "message": {
    "role": "user",
    "content": "任务描述"
  }
}

# 3. 获取任务状态
GET /tasks/{task-id}

# 4. 取消任务
POST /tasks/{task-id}/cancel

# 5. 订阅任务更新 (SSE)
GET /tasks/{task-id}/subscribe
```

## 关键特性

### 1. 隐私保护

A2A设计强调智能体之间的**不透明性**：

- Client Agent不需要知道Server Agent的内部实现
- Server Agent不需要知道Client Agent的用户详情
- 通信通过标准化的Task和Artifact进行

```
┌─────────────────────────────────────────────────────────────┐
│              A2A 隐私保护设计                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────┐              ┌─────────────┐             │
│   │  Client     │              │   Server    │             │
│   │   Agent     │              │   Agent     │             │
│   │             │              │             │             │
│   │ ┌─────────┐ │              │ ┌─────────┐ │             │
│   │ │ 内部状态 │ │◄── 不透明 ──►│ │ 内部状态 │ │             │
│   │ │ 用户数据 │ │    通信      │ │ 业务逻辑 │ │             │
│   │ └─────────┘ │              │ └─────────┘ │             │
│   │             │              │             │             │
│   │ 只暴露:     │              │ 只暴露:     │             │
│   │ • Task请求  │◄────────────►│ • 服务能力  │             │
│   │ • Artifact  │              │ • Task结果  │             │
│   └─────────────┘              └─────────────┘             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 安全机制

A2A支持多种安全认证方式：

| 认证方式 | 适用场景 | 安全级别 |
|---------|---------|---------|
| **OAuth 2.0** | 用户授权场景 | 高 |
| **API Keys** | 服务间通信 | 中 |
| **JWT Tokens** | 无状态认证 | 中 |
| **mTLS** | 高安全要求 | 最高 |

### 3. 异步与流式

A2A支持多种通信模式：

```
┌─────────────────────────────────────────────────────────────┐
│              A2A 通信模式                                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  同步模式                      异步流式模式                  │
│  ────────                      ───────────                  │
│                                                             │
│  Request ──► Response          Request ──► Ack              │
│                                                             │
│  适用: 快速查询                 适用: 长时间任务             │
│                                                             │
│  ──────────────────            ────────────────────────     │
│                                                             │
│  SSE流式推送                                                  │
│  ────────────                                               │
│                                                             │
│  Subscribe ──► Event 1 ──► Event 2 ──► ... ──► Complete    │
│                                                             │
│  适用: 实时进度更新                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 优势与挑战

### 优势

1. **跨厂商互操作**: 打破平台壁垒，实现真正的智能体互联
2. **标准化通信**: 统一的协议降低集成成本
3. **安全设计**: 内置隐私保护和安全机制
4. **灵活扩展**: 支持多种通信模式和认证方式
5. **Google生态**: 与Google Cloud、Vertex AI深度整合

### 挑战

1. **采用率**: 需要各厂商广泛支持
2. **复杂度**: 多智能体协调增加了系统复杂度
3. **信任建立**: 用户需要信任第三方智能体
4. **责任归属**: 多智能体协作时的责任界定
