# UCP (Universal Commerce Protocol) 深度解析

## 一句话定义

UCP 是 Google 于 2025 年初推出的开放标准协议，为 AI 智能体与商家系统之间提供标准化的商务交互框架。它采用**服务器选择协议（Server-Select Protocol）**架构，通过反向域名命名空间组织，实现支付工具与处理程序的解耦，解决 N×N 复杂性问题。

---

## 核心概念（Core Concepts）

UCP 的设计围绕以下核心概念构建，这些概念构成了协议的基础架构：

### 1. 服务器选择协议（Server-Select Protocol）

UCP 采用独特的服务器选择机制，不同于传统的客户端调用模式：

```
传统协议：                        UCP 服务器选择：
客户端 ──► 服务器                 平台 ──► 发现 ──► 商家服务器
（直接调用）                      （基于能力匹配选择）
```

**关键特性**：
- **反向域名命名空间**：使用 `com.example.merchant` 格式标识商家
- **能力协商**：平台必须计算平台能力与商家能力的交集
- **解耦设计**：支付工具与处理程序分离，解决 N×N 复杂性

### 2. Discovery（发现机制）

UCP 使用标准化的发现机制定位商家服务：

```
┌─────────────────────────────────────────────────────────────┐
│                    UCP 发现流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Well-Known Endpoint                                     │
│     商家在 `/.well-known/ucp` 托管 Profile 文件             │
│                                                             │
│  2. Profile 内容                                            │
│     {                                                       │
│       "services": [                                         │
│         {                                                   │
│           "id": "checkout",                                 │
│           "capabilities": ["payment", "shipping"],          │
│           "handlers": {                                     │
│             "payment": "google_pay",                        │
│             "shipping": "fedex"                             │
│           }                                                 │
│         }                                                   │
│       ]                                                     │
│     }                                                       │
│                                                             │
│  3. 能力匹配                                                │
│     平台计算：平台能力 ∩ 商家能力 = 可用功能                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Profile 包含**：
- **Services**：商家提供的服务列表
- **Capabilities**：每个服务支持的能力
- **Handlers**：处理程序配置（支付方式、物流等）
- **Extensions**：扩展能力定义

### 3. Capabilities & Handlers（能力与处理程序）

UCP 将商务功能抽象为**能力（Capabilities）**和**处理程序（Handlers）**两层：

| 层级 | 定义 | 示例 | 特点 |
|------|------|------|------|
| **Capabilities** | 声明式功能接口 | `checkout`、`inventory`、`shipping` | 标准化、版本独立 |
| **Handlers** | 具体实现配置 | `google_pay`、`stripe`、`fedex` | 可插拔、商家自选 |

**能力版本管理**：
- 使用 `YYYY-MM-DD` 日期格式版本号
- 商家必须支持旧版本平台请求
- 能力独立版本控制

### 4. Extensions（扩展机制）

UCP 支持通过扩展机制定制功能：

```json
// 基础 Schema
{
  "type": "checkout",
  "version": "2025-01-15"
}

// 扩展 Schema（使用 allOf 组合）
{
  "allOf": [
    { "$ref": "#/checkout-base" },
    {
      "properties": {
        "loyalty_program": { "type": "string" },
        "custom_tax_rules": { "type": "array" }
      }
    }
  ]
}
```

**扩展解析**：
- 平台在客户端解析扩展 Schema
- 使用 JSON Schema 的 `allOf` 组合机制
- 商家可添加自定义字段

## 支付资产管理（Payment Asset Management）

UCP 的核心创新之一是**支付工具与处理程序的解耦**，实现精细化的支付资产管理：

### 信任三角（Trust Triangle）

```
┌─────────────────────────────────────────────────────────────┐
│                    UCP 信任三角                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    ┌─────────────┐                          │
│                    │   商家      │                          │
│                    │  (Business) │                          │
│                    └──────┬──────┘                          │
│                           │                                 │
│              处理支付请求   │   提供商品/服务                │
│                           │                                 │
│     ┌─────────────────────┼─────────────────────┐           │
│     │                     │                     │           │
│     │                     ▼                     │           │
│     │            ┌─────────────┐                │           │
│     │            │    平台     │                │           │
│     └───────────►│ (Platform)  │◄───────────────┘           │
│       返回令牌   └──────┬──────┘   发起支付                  │
│                         │                                   │
│                         │ 获取凭证                           │
│                         ▼                                   │
│                  ┌─────────────┐                            │
│                  │  凭证提供商  │                            │
│                  │ (Credential │                            │
│                  │  Provider)  │                            │
│                  └─────────────┘                            │
│                                                             │
│  关键：三方之间不共享原始密钥，通过令牌间接通信              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 支付流程三阶段

UCP 将支付流程标准化为三个阶段：

#### 阶段 1：Advertisement（广告/声明）

商家向平台声明支持的支付方式：

```json
{
  "handlers": {
    "payment": {
      "type": "google_pay",
      "tokenizer": "stripe",
      "supported_networks": ["visa", "mastercard"],
      "supported_auth_methods": ["pan_only", "cryptogram_3ds"]
    }
  }
}
```

#### 阶段 2：Acquisition（获取）

平台直接与凭证提供商交互，获取支付令牌：

```
流程：
1. 平台识别商家支持的支付方式
2. 平台直接与 Google Pay / Stripe 交互
3. 凭证提供商返回加密令牌
4. 商家**从未接触**原始 PAN（主账号）
```

**PCI 范围最小化**：
- 商家不处理原始卡号
- 平台可选择仅处理令牌
- 大幅降低合规成本

#### 阶段 3：Completion（完成）

平台提交不透明令牌，商家后端完成扣款：

```
商家后端 ──► 支付网关（Stripe/Adyen）
     │
     └── 提交：加密令牌 + 金额
     
支付网关 ──► 解密并扣款
     │
     └── 返回：支付结果
```

### 支付资产类型

UCP 支持多种支付资产类型：

| 资产类型 | 说明 | 适用场景 |
|----------|------|----------|
| **Tokenized Cards** | 令牌化银行卡 | 标准信用卡支付 |
| **Saved Credentials** | 保存的支付凭证 | 一键支付 |
| **Digital Wallets** | 数字钱包（Google Pay/Apple Pay） | 移动支付 |
| **Cryptocurrency** | 加密货币 | Web3 场景 |
| **Bank Transfers** | 银行转账 | 大额支付 |

## 线上全链路流程（End-to-End Flow）

### 完整交易生命周期

```
┌─────────────────────────────────────────────────────────────┐
│              UCP 完整交易生命周期                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Phase 1: Discovery（发现）                                 │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  AI Agent ──► 平台                                   │    │
│  │  "用户想买跑鞋"                                       │    │
│  │                                                      │    │
│  │  平台 ──► 商家 Discovery Endpoint                     │    │
│  │  GET /.well-known/ucp                                │    │
│  │                                                      │    │
│  │  商家返回 Profile（能力、处理程序、扩展）             │    │
│  └─────────────────────────────────────────────────────┘    │
│                          │                                  │
│                          ▼                                  │
│  Phase 2: Negotiation（协商）                               │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  平台计算能力交集                                     │    │
│  │  Platform Capabilities ∩ Merchant Capabilities      │    │
│  │                                                      │    │
│  │  确定可用功能：                                       │    │

│                          │                                  │
│                          ▼                                  │
│  Phase 3: Shopping（购物）                                  │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  AI Agent ──► 平台 ──► 商家                          │    │
│  │                                                      │    │
│  │  1. 语义搜索："适合扁平足的跑鞋"                     │    │

│                          │                                  │
│                          ▼                                  │
│  Phase 4: Checkout（结账）                                  │

│  │  方式 A：嵌入式结账（Embedded Checkout）              │    │
│  │  • 商家结账界面嵌入平台                               │    │
│  │  • 支持人机协作（ECP 协议）                           │    │
│  │                                                      │    │
│  │  方式 B：API 结账                                     │    │
│  │  • 纯程序化交互                                       │    │

│  └─────────────────────────────────────────────────────┘    │
│                          │                                  │
│                          ▼                                  │
│  Phase 5: Payment（支付）                                   │
│  ┌─────────────────────────────────────────────────────┐    │

│                          ▼                                  │

│  ┌─────────────────────────────────────────────────────┐    │
│  │  • 订单状态同步                                       │    │
│  │  • 物流追踪                                           │    │
│  │  • 交付确认                                           │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 状态机与交接（State Machine & Handoff）

UCP 使用标准化状态机管理交易状态：

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ DISCOVER│────►│ NEGOTIATE│────►│  SHOP   │────►│ CHECKOUT│
└─────────┘     └─────────┘     └────┬────┘     └────┬────┘
                                     │               │
                                     ▼               ▼
                              ┌─────────┐     ┌─────────┐
                              │  ABORT  │     │  PAY    │
                              │ (可选)  │     └────┬────┘
                              └─────────┘          │
                                                   ▼
                                            ┌─────────┐
                                            │COMPLETE │
                                            └─────────┘
```

**状态说明**：
- **DISCOVER**：发现商家服务
- **NEGOTIATE**：能力协商
- **SHOP**：商品浏览、选择
- **CHECKOUT**：结账准备
- **PAY**：支付处理
- **COMPLETE**：交易完成
- **ABORT**：交易中止

## 嵌入式结账协议（ECP - Embedded Checkout Protocol）

UCP 的 ECP 实现了**人机协作**的结账体验：

### ECP 核心机制

```
┌─────────────────────────────────────────────────────────────┐
│                    ECP 架构                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐         postMessage         ┌─────────────┐│
│  │    Host     │◄───────────────────────────►│   Merchant  ││
│  │  (Platform) │      JSON-RPC 2.0           │   iFrame    ││
│  │             │                             │             ││
│  │ • Google    │                             │ • 结账界面  ││
│  │ • Shopify   │                             │ • 支付表单  ││
│  │ • 自建平台  │                             │ • 地址选择  ││
│  └──────┬──────┘                             └──────┬──────┘│
│         │                                           │       │
│         │  任务委托                                  │       │
│         │  (如：选择地址)                            │       │
│         ▼                                           ▼       │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              人机协作流程                            │    │
│  │                                                      │    │
│  │  Agent 可以：                                        │    │
│  │  • 自动填充已知信息（姓名、地址）                    │    │
│  │  • 推荐商品选项                                      │    │
│  │                                                      │    │
│  │  人工介入场景：                                      │    │
│  │  • 确认支付金额                                      │    │
│  │  • 选择支付方式                                      │    │
│  │  • 输入安全验证码                                    │    │
│  │                                                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### ECP 状态通知

商家通过通知机制同步状态变化：

| 通知类型 | 触发条件 | 说明 |
|----------|----------|------|
| `ec.ready` | 结账准备就绪 | 初始化完成，可以开始交互 |
| `ec.start` | 结账界面可见 | 用户进入结账流程 |
| `ec.line_items.change` | 购物车内容变更 | 商品增删改 |
| `ec.buyer.change` | 买家信息变更 | 地址、联系方式更新 |
| `ec.complete` | 订单成功创建 | 交易完成 |
| `ec.message.change` | 错误或警告更新 | 显示校验错误 |

### ECP 安全要求

- **Origin 验证**：验证消息来源
- **CSP 配置**：设置 `frame-ancestors` 限制嵌入
- **PUT 风格更新**：整对象替换确保单一真相源

## 与 AP2 的深度集成

UCP 与 AP2（Agent Payments Protocol）形成互补关系：

### 分工架构

```
┌─────────────────────────────────────────────────────────────┐
│                 UCP + AP2 集成架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  UCP 负责：商务层（What & Where）                           │
│  ├─ 商品发现与选择                                          │
│  ├─ 购物车管理                                              │
│  ├─ 结账流程                                                │
│  └─ 订单履约                                                │
│                                                             │
│  AP2 负责：支付层（How）                                    │
│  ├─ 支付授权（Mandate）                                     │
│  ├─ 身份验证                                                │
│  ├─ 资金结算                                                │
│  └─ 审计追踪                                                │
│                                                             │
│  集成点：                                                   │
│  UCP Checkout ──► AP2 Mandate ──► Payment Completion       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Mandate 在 UCP 中的应用

AP2 的 Mandate 机制为 UCP 提供支付授权：

```
1. 用户预先授权（AP2 Mandate）
   "允许 AI 在 100 美元以内帮我购买书籍"
   
2. UCP 购物流程
   AI 发现书籍 → 加入购物车 → 发起结账
   
3. 支付时验证 Mandate（AP2）
   检查：金额 ≤ 100 美元 ✓
   检查：商品类别 = 书籍 ✓
   检查：时间窗口有效 ✓
   
4. 完成支付
   使用 AP2 授权令牌完成 UCP 结账
```

## 风险分析与安全机制

### 风险矩阵

| 风险类型 | 风险描述 | UCP 缓解措施 |
|----------|----------|--------------|
| **支付欺诈** | 未经授权的交易 | Mandate 范围限制、多因素认证 |
| **数据泄露** | 敏感支付信息泄露 | 令牌化、信任三角架构 |
| **身份冒用** | Agent 冒充用户 | 可验证凭证（VC）签名 |
| **中间人攻击** | 通信被截获篡改 | HTTPS、消息签名验证 |
| **商家欺诈** | 虚假商家收款 | Profile 验证、信誉系统 |

### 安全机制详解

#### 1. 可验证凭证（Verifiable Credentials）

```
用户身份 ──► 可验证凭证（VC）──► 数字签名
     │                              │
     │                              ▼
     │                         防篡改证明
     │                              │
     └──────────────────────────────┘
            商家验证签名真实性
```

#### 2. PCI 合规优化

UCP 通过架构设计最小化 PCI DSS 范围：

| 参与方 | 接触数据 | PCI 范围 |
|--------|----------|----------|
| 商家 | 仅令牌 | 最小化/免除 |
| 平台 | 可选令牌 | 可控 |
| 凭证提供商 | 原始 PAN | 完全合规 |

#### 3. 审计追踪

完整交易链路可审计：

```
Intent ──► Discovery ──► Negotiation ──► Shopping 
  │                                               │
  └──────────────── 审计日志记录 ─────────────────┘
  
Checkout ──► Payment ──► Fulfillment ──► Complete
  │                                               │
  └──────────────── 审计日志记录 ─────────────────┘
```

## 协议对比：UCP vs ACP

| 维度 | UCP | ACP |
|------|-----|-----|
| **定位** | 完整商务协议 | 支付授权协议 |
| **覆盖范围** | 发现 → 选择 → 购买 → 履约 | 授权 → 支付 |
| **架构** | 服务器选择协议 | 委托令牌协议 |
| **发现机制** | Well-Known Endpoint | 商家注册 |
| **支付处理** | 信任三角 + 令牌化 | 委托支付令牌 |
| **人机协作** | ECP 嵌入式结账 | 实时确认 |
| **生态依赖** | Google 主导 | OpenAI + Stripe |
| **开放性** | 开放标准，多平台支持 | 开放标准，跨平台 |
| **最佳场景** | 复杂购物决策 | 快速支付结账 |

## 总结

UCP 代表了 AI 商务基础设施的完整解决方案：

1. **架构创新**：服务器选择协议解决 N×N 复杂性
2. **安全设计**：信任三角实现支付工具与处理程序解耦
3. **全链路覆盖**：从发现到履约的完整商务生命周期
4. **人机协作**：ECP 协议支持无缝的人工介入
5. **生态整合**：与 AP2、A2A、MCP 等协议形成互补

它是 Google 构建"AI 操作系统"在商务领域的核心布局，也是未来智能购物的基础设施标准。
