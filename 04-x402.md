# x402 协议深度调研报告

## 概述

**x402** 是由 **Coinbase** 推出的支付协议，旨在**"使价值像信息一样无缝跨越互联网"**。x402的核心创新是激活了HTTP协议中长期被保留但从未广泛使用的 **402 Payment Required** 状态码，为AI智能体时代的价值交换提供了原生解决方案。

> **发布时间**: 2024-2025年  
> **主要推动方**: Coinbase  
> **协议类型**: 互联网原生支付协议  
> **开源状态**: 完全开源 ([GitHub](https://github.com/coinbase/x402))

---

## 核心愿景

x402的愿景是构建一个**互联网原生的支付层**，让价值交换像信息交换一样简单：

```
┌─────────────────────────────────────────────────────────────┐
│              x402: 让价值像信息一样流动                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   信息互联网                    价值互联网                   │
│   ──────────                    ──────────                  │
│                                                             │
│   HTTP GET ──► 200 OK ──► 内容    vs    HTTP GET ──► 402 ──► 支付 ──► 200  │
│                                                             │
│   [请求]───►[免费获取]───►[数据]   vs   [请求]───►[需要支付]───►[💰]───►[资源] │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### HTTP 402 状态码的历史

HTTP 402 (Payment Required) 是HTTP/1.1标准中预留的状态码，但几十年来从未被正式使用。x402终于赋予了这个状态码实际意义：

## 技术架构

### 协议规范

```yaml
# x402 协议规范
protocol: x402
version: "2.0"
base: HTTP/1.1

# 核心状态码
status_codes:
  402: Payment Required
  200: OK (payment successful)
  403: Forbidden (payment failed)

# 请求头
headers:
  X-402-Price: "支付金额"
  X-402-Asset: "支付资产(如USDC)"
  X-402-Address: "收款地址"
  X-402-Network: "区块链网络"

# 响应头
response_headers:
  X-402-Receipt: "支付收据"
  X-402-Transaction: "交易哈希"
```

### x402 V2 升级

x402 V2引入了重要改进：

| 特性 | V1 | V2 |
|------|-----|-----|
| **多链支持** | 单链 | 统一接口支持多链 |
| **身份验证** | 地址验证 | 基于钱包的身份 |
| **会话管理** | 每次请求都需支付 | 可重用会话减少延迟 |
| **支付模式** | 按请求 | 按请求 + 订阅模式 |

## 使用场景

### 1. API按需付费

取代传统的API Key订阅模式：

```javascript
// 传统方式: 月度订阅
const apiKey = "sk-xxx";
const response = await fetch("https://api.example.com/data", {
  headers: { "Authorization": `Bearer ${apiKey}` }
});

// x402方式: 按调用付费
const response = await fetch("https://api.example.com/data");
// 如果返回402，客户端自动处理支付并重试
// 每次调用只支付实际使用的部分
```

### 2. 付费内容访问

AI智能体可以按需解锁付费内容：

```
用户: "帮我查找关于量子计算的最新研究"

      ↓
```

## 与ACP/AP2的对比

| 维度 | x402 | AP2 | ACP |
|------|------|-----|-----|
| **主要推动方** | Coinbase | Google | OpenAI + Stripe |
| **支付模式** | 按请求微支付 | 基于凭证授权 | 基于购物车 |
| **结算方式** | 区块链/稳定币 | 传统+数字支付 | 传统支付 |
| **技术基础** | HTTP 402 | REST API | REST API |
| **去中心化** | 高 | 中 | 低 |
| **最佳场景** | API调用、内容付费 | 电商、服务采购 | 电商购物 |
| **延迟** | 区块链确认时间 | 实时 | 实时 |

## 优势与挑战

### 优势

1. **互联网原生**: 基于HTTP标准，无需额外基础设施
2. **去中心化**: 区块链结算，无需信任第三方
3. **按需付费**: 精确的按使用付费模式
4. **全球可达**: 不受地域和银行系统限制
5. **低门槛**: 任何支持HTTP的客户端都可以使用

### 挑战

1. **区块链延迟**: 交易确认需要时间
2. **gas费用**: 小额支付可能不经济
3. **监管不确定**: 加密货币支付面临监管挑战
4. **用户教育**: 需要用户理解加密钱包等概念
